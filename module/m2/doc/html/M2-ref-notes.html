Project Lupine — Development Roadmap

The development roadmap for Project Lupine essentially parallels the structure of this document.

  1. Develop UML Infrastructure Library implementation

  2. Develop MOF (M3) implementation

  3. Develop UML Superstructure implementation

  4. Develop XMI model serialization framework

  5. Develop Tioga IDE Integration for Project Lupine components (e.g CLIM presentations for MOF and UML entities)

  6. Develop Lupine M2 Metamodel for Common Lisp

  7. Develop other “To Do” items onto Tioga IDE, cf. Tioga IDE

N.b, Project Lupine is a non-commercial project. There is not a formal schedule available, at this time, for the completion of the preceding roadmap.

Sidebar: Lupine M2 Metamodel for Common Lisp
Project Lupine proposes to develop an additional metamodel for modeling of Common Lisp program elements. The development of that metamodel's formal specification in Project Lupine may be preceded by one or more prototypes in the form of, each, a UML profile for modeling of selected subsets of Common Lisp program elements, in compatible modeling platforms.

UML

UML Metamodel

The UML metamodel is defined effectively by the UML infrastructure specification [UML Infrastructure 2.4.1], collectively with the UML Superstructure specification [UML Superstructure 2.4.1]. For applications requiring a formal, machine-readable format for interpretation of the structures defined in the UML metamodel, the UML metamodel is published as serialized in XMI format [XMI 2.4.1] by the OMG.

UML Profiles

A UML profile provides a means for extending the UML metamodel with domain-specific features, within a UML model. In a sense of the semantics of the four-layered modeling architecture, a UML profile may serve effectively as a bridge between the UML user model (M1) and the metamodel modeling layer (M3), such as demonstrated in the SPEM 2.0 profile for UML. Alternately, a UML profile may extend on features defined specifically in the UML metamodel (M2), for modeling of domain-specific entities.

UML Associations

A UML association describes a UML relationship among one or more UML types.

Binary Associations

N-ary Associations

Directedness and Navigability

The formal semantics of UML Infrastructure [UML Infrastructure 2.4.1] do not require that an association be defined as a directed relation. An association in UML defines, essentially, a relation among properties in the model.

Directedness in a UML association may be implied of the navigability of association member end points.

Annotation of UML Association Endpoints

  • Navigable Endpoint: “Lined arrow,” may be used for display of navigable endpoints (optional)

  • Composite Endpoint: “Hollow diamond,” used for n-ary associations with n>2

  • Aggregate Endpoint: “Filled diamond,” used for aggregate composite associations

  • Specialized association types: Defined in UML superstructure [UML Superstructure 2.4.1] (e.g generalization, realization, dependemcy)

Tioga IDE

The Tioga IDE. at present, is a “Blue Sky” project concept being developed within Project Lupine. As such, Tioga IDE represents a concept towards the development of a central IDE framework in Common Lisp. In an initial draft of the requirements for the Tioga IDE project, the Tioga IDE components for Project Lupine may be designed as to meet a set of requirements including the following, albeit in a “Blue Sky” regards:

  • Provide CLIM (cf. McCLIM) presentation methods for entities modeled within the Project Lupine MOF implementation

  • Provide graphical presentations for the SPEM 2.0 Profile for UML

  • Integrate with cross-platform object access protocols such as CL+J (cf. MKCL) for purpose of visual modeling of objects initialized within external object systems

  • Incorporate distributed object system protocols (e.g CORBA) and transport-layer security (TLS) measures for access to remote model repositories from within the Tioga IDE desktop workspace

  • Provide graphical presentations of systems simulation models extending on metsmodels supported within Project Lupine

  • Provide graphical controls for the Hypertext Coffee Pot Control Protocol, ultimately incorporating I/O frameworks for I2C, SMBus, and other hardware integration technologies

  • Model MIT CADR, in the CADR Lisp Machine's own original hardware designs and later VHDL system/circuit representations, so far as may not be limited with applicable intellectual property law

  • Support multiple modeling platform vendor practices for XMI model serialization, hypothetically including support for:

    • NoMagic Inc's MagicDraw

    • IBM Rational Rose

    • Sparx Systems Enterprise Architect

    • Yatta UML Lab

    • Eclipse Papyrus

    • Modelio

    • ArgoUML (UML 1.4 compatibility)

    • Cubetto (Mobile and Desktop Editions)
